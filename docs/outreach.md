# ProofTask Outreach

## Goal

The goal of outreach is not to sell a big platform immediately.

The goal is to find 3-5 early teams that already use AI coding agents and feel uncertainty around review, QA, or release confidence.

Primary ask:

> Would it be useful to test ProofTask on one AI-generated pull request or feature?

## Ideal customer profile

### ICP 1: AI-heavy SaaS teams

These are small teams shipping product with AI coding tools.

Signals:

- team uses Cursor, Copilot, Codex, Claude Code, or similar tools;
- founders talk about AI-assisted development;
- team ships fast;
- QA process is lightweight or informal;
- team worries about regressions.

Best message:

> You are moving faster with AI coding tools. ProofTask helps you keep human verification evidence before merge or release.

### ICP 2: AI coding agent builders

These teams build agents that write code, open PRs, or automate development workflows.

Signals:

- product mentions AI agents, coding agents, autonomous engineering, PR automation, code generation;
- users need trust, review, approval, or audit flows;
- agent output must be accepted by humans.

Best message:

> Your agent can create work. ProofTask helps your users verify the work with structured human proof.

### ICP 3: QA agencies and freelance QA leads

These people already sell testing services and may need an AI-native workflow.

Signals:

- QA agency website;
- manual testing services;
- startup testing packages;
- crowdtesting or exploratory testing;
- LinkedIn posts about AI in QA.

Best message:

> ProofTask can turn AI-agent QA requests into structured tasks, proof packages, and client-ready trace artifacts.

### ICP 4: Open-source maintainers

These maintainers receive or expect more AI-generated pull requests.

Signals:

- active GitHub repo;
- many PRs;
- maintainers discuss review burden;
- project has AI-generated contributions;
- labels like `needs review`, `qa`, `regression`, `test required`.

Best message:

> ProofTask can help maintainers request human proof for agent-generated changes before merge.

### ICP 5: Regulated or high-risk software teams

These teams care about evidence, not just speed.

Signals:

- fintech, healthtech, infrastructure, security, compliance;
- mistakes are expensive;
- release decisions need evidence;
- audit trails matter.

Best message:

> ProofTask creates human verification traces for AI-generated work before it reaches production.

## Outreach rules

Keep the first message short.

Do not pitch a huge vision immediately.

Do not say ProofTask is a marketplace.

Do not ask for money in the first message unless there is clear buyer intent.

Ask for one concrete test:

> Can we try this on one AI-generated PR or feature?

## Message 1: founder using AI coding tools

```text
Hi {name},

I saw you are building with AI coding tools.

I am building ProofTask — a lightweight human verification layer for AI-generated software work.

When an AI agent creates a PR or feature, ProofTask turns the review into a structured QA task, collects human proof, and saves a traceable verification artifact before merge or release.

Would it be useful to test this on one AI-generated change in your workflow?
```

## Message 2: AI coding agent builder

```text
Hi {name},

I am exploring a verification layer for AI coding agents.

The idea: your agent creates work, then ProofTask creates a structured human verification task with acceptance criteria, required proof, and a final trace.

This could help users trust agent-created PRs before merge or release.

Would you be open to testing this around one agent-created PR or workflow?
```

## Discovery questions

Use these in calls or replies:

1. What AI coding tools are you using today?
2. Do AI tools create PRs or only assist developers locally?
3. Who reviews AI-generated changes?
4. What makes a PR safe to merge?
5. Do you require screenshots, test notes, or QA evidence?
6. Where does that evidence live today?
7. Have you had bugs from AI-generated work?
8. Would a structured proof artifact be useful?
9. Who would care most about this: founder, CTO, QA, compliance, or product?
10. Would you pay for a pilot if it reduced review uncertainty?

## Qualification checklist

A prospect is qualified if at least 3 are true:

- uses AI coding tools weekly;
- ships product frequently;
- has review or QA bottlenecks;
- cares about release confidence;
- uses GitHub pull requests;
- stores QA proof informally;
- worries about AI-generated mistakes;
- has a founder, CTO, or QA lead who can approve a pilot.

## Pilot call structure

### 1. Context

Ask how they use AI coding tools today.

### 2. Pain

Ask where review, QA, or proof breaks down.

### 3. Demo

Show the ProofTask flow:

```text
Agent task -> human proof -> verifier decision -> trace
```

### 4. Pilot proposal

Propose one narrow test:

> Let us verify one AI-generated PR or feature and produce a proof trace.

### 5. Decision

Ask:

> If this creates useful evidence, who would need to approve the next pilot?

## What not to do

Do not sell the marketplace first.

Do not over-explain the long-term vision.

Do not claim ProofTask replaces automated testing.

Do not promise full compliance.

Do not sell to everyone.

Do not ask for feedback from people who do not use AI coding tools.

## Success metric

The outreach campaign is successful if it produces:

- 30 targeted messages sent;
- 5-10 replies;
- 3 discovery calls;
- 1 paid or serious pilot conversation;
- 1 concrete AI-generated PR or feature to test.

## Weekly outreach sprint

A simple weekly sprint:

- Monday: find 20 prospects;
- Tuesday: send 10 messages;
- Wednesday: send 10 messages;
- Thursday: follow up and book calls;
- Friday: review replies and improve messaging;
- Weekend: write learnings into ProofTask docs.

## Positioning reminder

ProofTask is not selling tasks.

ProofTask is selling confidence:

> AI agents can move fast. ProofTask helps humans verify what matters.
